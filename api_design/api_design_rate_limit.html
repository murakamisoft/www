APIのレート制限（rate limit）設計方法

APIのレート制限は「システムを守る防御設計」です。過負荷・攻撃・バグ暴走を防ぐための重要な仕組みを実務レベルで解説します。

1. レート制限とは何か

レート制限とは「一定時間内にAPIを呼び出せる回数を制御する仕組み」です。

項目 内容
目的 過負荷防止・不正アクセス防止
単位 秒・分・時間あたりのリクエスト数

2. なぜ必要か

問題① DDoS攻撃対策

大量リクエストによるサービス停止を防ぐ。

問題② バグ暴走防止

ループ処理や誤実装による過負荷を防止。

問題③ コスト制御

クラウドAPIの従量課金を抑制する。

3. 設計パターン

3-1. 固定ウィンドウ方式

一定時間ごとにリセットする方式。

3-2. スライディングウィンドウ方式

より正確に制御できる方式。

3-3. トークンバケット方式

柔軟性が高く実務でよく使われる。


1秒あたり100リクエストまで許可

4. Spring Boot実装例


@RestController
public class RateLimitController {

    private int count = 0;
    private final int LIMIT = 100;

    @GetMapping("/api")
    public String api() {

        if (count >= LIMIT) {
            return "TOO MANY REQUESTS";
        }

        count++;
        return "OK";
    }
}

実務ではRedisやAPI Gatewayで制御するのが一般的です。

5. よくあるエラー

制限が効かない

複数サーバー構成でカウントが共有されていない。

対処法

  • Redisでカウンタ共有
  • API Gateway導入
  • トークンバケット方式採用

6. まとめ

  • レート制限はAPI防御の基本
  • 単体サーバー実装は本番NG
  • Redis・Gatewayでの制御が必須
