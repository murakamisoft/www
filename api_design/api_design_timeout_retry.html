APIのタイムアウト設計とリトライ戦略

1. タイムアウト設計とは

タイムアウトとは「一定時間で処理を強制終了する設計」です。

項目 内容
目的 無限待機の防止・障害波及防止
対象 API・DB・外部サービス

2. リトライ設計の考え方

やってはいけない例

  • 無制限リトライ
  • 即時リトライ連打

正しい考え方

  • 回数制限（例：3回まで）
  • 指数バックオフ（待機時間増加）
  • 対象エラー限定（5xxのみ）

3. 設計パターン

3-1. 固定リトライ

一定間隔で再試行するシンプル方式。

3-2. 指数バックオフ

リトライごとに待機時間を増やす方式。


1秒 → 2秒 → 4秒 → 8秒

3-3. サーキットブレーカー連携

障害時に自動遮断する高度設計。

4. Spring Boot実装例


public String callApi() {

    int retry = 0;

    while (retry < 3) {
        try {
            return externalService.call();
        } catch (Exception e) {

            retry++;

            try {
                Thread.sleep(1000L * retry);
            } catch (InterruptedException ignored) {}
        }
    }

    throw new RuntimeException("API failed after retry");
}

実務ではResilience4jやSpring Retryを使うのが一般的です。

5. よくあるエラー

無限リトライ

障害時にシステム全体を圧迫する。

対処法

  • 最大リトライ回数を設定
  • 指数バックオフ導入
  • サーキットブレーカー併用

6. まとめ

  • タイムアウトは必須の防御設計
  • リトライは制御しないと危険
  • 外部依存は必ず失敗前提で設計
