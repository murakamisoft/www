よくあるAPI設計ミスまとめ

REST API設計で現場エンジニアがやりがちなミスと、その改善方法を実務レベルで整理します。 レガシー改善や設計レビューでそのまま使える内容です。

1. 概要

API設計の品質はシステム全体の保守性・拡張性に直結する。 特に初期設計のミスは後から修正コストが非常に高い。

対象 内容
環境 Java 21 / Spring Boot 3.x
想定読者 中級エンジニア・設計レビュー担当

2. よくある設計ミス

① 動詞ベースのURL設計

悪い例：/getUser /createUser

問題：RESTの「リソース設計」に違反し、統一性が崩れる


# NG
GET /getUser?id=1
POST /createUser

# OK
GET /users/1
POST /users

② HTTPメソッドの誤用

POSTで取得、GETで更新などの設計ミス

影響：キャッシュ・セキュリティ・監視が破綻

③ ステータスコードを全部200にする

フロント側がエラー判定できず、バグ調査コストが増大する


# NG
{ "status": "error" }

# OK
HTTP 400 + JSONエラー

④ レスポンス構造がバラバラ

プロジェクトごとにJSON形式が違うケース

→ フロント実装が複雑化し工数増加

3. 改善例（Before / After）

Before After
/getUser?id=1 /users/1
/createOrder POST /orders

REST設計の基本に統一するだけで、保守性は大幅に改善する。

4. よくあるエラー・事故パターン

① null返却によるフロントクラッシュ

対策：空配列 or 明示的エラー返却

② 想定外のステータスコード

対策：API仕様書でコードを固定化

③ API変更による互換性崩壊

対策：バージョニング（/v1 /v2）

5. まとめ

  • URLはリソース中心で設計する
  • HTTPメソッドとステータスコードを厳密に使う
  • レスポンス形式を統一する

API設計のミスは早期に潰すほどコストが下がる。 レビュー時点で標準化することが重要。
