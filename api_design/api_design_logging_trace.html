APIのログ設計は「障害が起きたときに原因を追えるか」を決める重要な設計です。トレースIDを中心に実務レベルで解説します。

1. ログ設計とは何か

ログ設計とは、システムの動作記録を「あとから追える形で残す設計」です。

項目 内容
対象 API・バッチ・非同期処理
目的 障害調査・監視・分析

2. なぜ重要か

問題① 障害原因が追えない

ログが不足していると原因特定に時間がかかる。

問題② 分散システムで追跡不能

複数API間でリクエストが分断される。

問題③ 運用コスト増大

調査に時間がかかり運用負荷が増える。

3. 正しい設計パターン

3-1. トレースID設計

すべてのリクエストに一意IDを付与します。


traceId: 7f3a9c2b-xxxx

3-2. ログ出力レベル設計

  • INFO：正常処理
  • WARN：異常の予兆
  • ERROR：障害

3-3. 構造化ログ


{
  "traceId": "123",
  "userId": "u001",
  "action": "login"
}

4. Spring Boot実装例


@RestController
public class LogController {

    @GetMapping("/test")
    public String test() {

        String traceId = UUID.randomUUID().toString();

        System.out.println("[TRACE]" + traceId + " start");

        return "OK";
    }
}

実務ではMDC（Mapped Diagnostic Context）を使って自動付与するのが一般的です。

5. よくあるエラー

ログが分断される

traceIdがないため関連ログが追えない。

対処法

  • 全リクエストにtraceId付与
  • MDC導入
  • 構造化ログ採用

6. まとめ

  • ログ設計は障害対応の生命線
  • traceIdは必須
  • 構造化ログで運用効率が大幅改善
