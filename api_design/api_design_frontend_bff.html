API設計とフロント連携のベストパターン（BFF含む）

フロントエンドとAPIは「そのまま直結」すると破綻します。本記事ではBFF（Backend for Frontend）を含めた実務設計パターンを解説します。

1. BFFとは何か

BFF（Backend for Frontend）とは、フロント専用に用意するAPI層です。 モバイル・WebでAPIを分離することで設計が安定します。

構成 役割
Frontend 画面表示・UI制御
BFF フロント用API集約・整形
Backend API ビジネスロジック

2. なぜAPI直結は危険か

問題① レスポンスが肥大化

フロントごとに必要なデータが違うため無駄が増える。

問題② 変更に弱い

API変更が全画面に影響する構造になる。

問題③ ビジネスロジック混入

フロント側でデータ加工が増え保守不能になる。

3. 正しい設計パターン

3-1. BFF分離構成


Frontend → BFF → Backend API

3-2. API統合パターン

複数APIをBFFでまとめて返す設計。


public class DashboardResponse {
    private User user;
    private List orders;
}

4. Spring Boot実装例


@RestController
@RequestMapping("/bff")
public class DashboardBffController {

    @GetMapping("/dashboard")
    public DashboardResponse getDashboard() {

        User user = userService.getUser();
        List orders = orderService.getOrders();

        return new DashboardResponse(user, orders);
    }
}

BFF層でAPIを集約することで、フロントは1リクエストで完結します。

5. よくあるエラー

BFFを作らず肥大化するケース


Frontend API = Backend API（直結）

→ 結果としてAPIが画面依存になり破綻します。

対処法

BFF層を必ず挟み、責務を分離すること。

6. まとめ

  • BFFはフロント依存を吸収する層
  • API直結は長期運用で破綻しやすい
  • Spring Bootで簡単に実装可能
