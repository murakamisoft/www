1. REST API設計の基本思想

REST APIは「リソース（資源）をURLで表現し、HTTPメソッドで操作する」という設計思想に基づく。 重要なのは「動詞ではなく名詞で設計する」点である。

  • URLはリソース（例：/users /orders）
  • 操作はHTTPメソッドで表現
  • ステートレス（状態をサーバーに持たない）

2. エンドポイント設計ルール

エンドポイントは一貫性が最も重要。


# 良い例
GET /users
GET /users/1
POST /users

# 悪い例
GET /getUser
POST /createUser
  • 動詞をURLに入れない
  • 複数形を基本にする（/users）
  • 階層構造でリソース関係を表す

3. HTTPメソッドの正しい使い方

メソッド 用途
GET データ取得
POST 新規作成
PUT 全更新
PATCH 部分更新
DELETE 削除

4. ステータスコード設計

ステータスコードはAPI品質を決める重要要素。

  • 200：正常終了
  • 201：作成成功
  • 400：リクエスト不正
  • 401：認証エラー
  • 404：リソースなし
  • 500：サーバーエラー

5. エラーレスポンス設計

エラーは必ず構造化する。


{
  "timestamp": "2026-06-12T10:00:00Z",
  "status": 400,
  "error": "Bad Request",
  "message": "userId is required",
  "path": "/users"
}

フロント側が解析しやすい形式に統一することで保守性が大きく向上する。

6. まとめ

  • URLはリソース中心で設計する
  • HTTPメソッドを正しく使い分ける
  • ステータスコードとエラー設計を統一する

REST API設計は「一貫性」がすべてであり、チーム開発では特に標準化が重要になる。
