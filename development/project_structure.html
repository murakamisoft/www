1. フォルダ構成の基本原則

フォルダ構成は「機能単位」で分けるのが基本です。クラスの役割（Controller / Service / Repository）ごとに分離し、責務を明確にします。

  • 責務が混ざらない構造にする
  • 機能単位 or ドメイン単位で分割する
  • 変更影響範囲を最小化する

2. よくある失敗パターン

  • Controller / Service が1フォルダに混在
  • 機能ごとではなく技術ごとに分割しすぎる
  • 共通処理が散らばる

この構造だと変更時に複数フォルダを横断する必要があり、保守性が低下します。

3. 推奨ディレクトリ構成


com.example.app
 ├── user
 │     ├── controller
 │     ├── service
 │     ├── repository
 │     └── dto
 ├── order
 │     ├── controller
 │     ├── service
 │     ├── repository
 │     └── dto
 └── common

機能単位（ドメイン単位）で分割することで、変更影響が局所化されます。

4. Spring Bootでの実践例


@RestController
@RequestMapping("/users")
public class UserController {

  private final UserService userService;

  public UserController(UserService userService) {
    this.userService = userService;
  }

  @GetMapping("/{id}")
  public User getUser(@PathVariable Long id) {
    return userService.findById(id);
  }
}

ControllerはHTTP処理のみ、ビジネスロジックはServiceに分離します。

5. まとめ

  • フォルダ構成は機能単位が基本
  • 技術単位分割はスケールで破綻しやすい
  • 責務分離が保守性を決める
