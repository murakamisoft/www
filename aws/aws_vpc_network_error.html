VPC設定ミスで通信できない原因まとめ

AWSのVPC環境でEC2へ接続できない、インターネットに出られない、サーバー間通信ができない場合の原因と対処法をまとめます。

1. VPC通信トラブルの概要

AWSでネットワーク障害が発生した場合、多くはVPC関連設定の不備が原因です。 特にルートテーブル、セキュリティグループ、NACL(Network ACL)の設定ミスが頻発します。

通信障害の切り分けでは、通信経路を上流から順番に確認することが重要です。

確認項目 内容
VPC CIDR設定が正しいか
サブネット 適切なAZ・CIDRになっているか
ルートテーブル 通信経路が定義されているか
Security Group 必要ポートが許可されているか
NACL 通信を拒否していないか
Internet Gateway インターネット接続設定があるか
NAT Gateway プライベートサブネット用に設定されているか

2. 確認すべき設定項目

2-1. ルートテーブルの確認

EC2からインターネットへ接続できない場合、まずルートテーブルを確認します。 


Destination      Target
0.0.0.0/0        igw-xxxxxxxx
10.0.0.0/16      local

インターネット接続には0.0.0.0/0がInternet Gatewayへ向いている必要があります。

2-2. Security Groupの確認

SSHやWebアクセスができない場合はSecurity Groupを確認します。


Type      Port
SSH       22
HTTP      80
HTTPS     443

必要ポートが許可されていないと通信できません。

2-3. NACL(Network ACL)の確認

Security Groupが正しくてもNACLで拒否されているケースがあります。

特にアウトバウンドルールが閉じられている場合は通信できなくなります。

2-4. NAT Gatewayの確認

プライベートサブネットのEC2は直接インターネットへ接続できません。

OSアップデートや外部API接続を行う場合はNAT Gatewayが必要です。

3. 通信確認コマンド例

EC2へログインできる場合は通信状況を確認します。


ping 8.8.8.8

curl https://www.google.com

nslookup amazon.com

traceroute 8.8.8.8

DNSだけ失敗しているのか、ネットワーク全体が失敗しているのかを切り分けられます。

4. よくあるネットワークエラー

EC2へSSH接続できない


Connection timed out

主な原因は以下です。

  • Security Groupで22番ポート未許可
  • パブリックIP未割当
  • Internet Gateway未設定
  • ルートテーブル設定ミス

インターネットへ接続できない


curl: (7) Failed to connect

NAT GatewayやInternet Gatewayの設定を確認してください。

VPC間通信できない


Request timeout

VPC PeeringやTransit Gateway設定、ルートテーブル設定漏れを確認します。

5. まとめ

  • 通信障害の多くはルートテーブル設定ミス
  • Security GroupとNACLは両方確認する
  • プライベートサブネットにはNAT Gatewayが必要
  • Internet Gateway未設定は頻出トラブル
  • 通信経路を上流から順番に確認する
