volumeマウントが効かない原因と解決方法

Dockerでvolumeマウントが反映されない、ファイルが見えない、データが消える場合の原因と対処法を解説します。

1. volumeマウントが効かない主な原因

Dockerのvolume関連トラブルは開発環境・本番環境ともによく発生します。 特にDocker Desktop利用時やDocker Compose利用時に起こりやすい問題です。

原因 内容
パス指定ミス ホスト側ディレクトリが間違っている
権限不足 コンテナからアクセスできない
匿名Volume 想定外のVolumeが生成される
Compose設定ミス マウント先の記述誤り
キャッシュ問題 変更が即時反映されない

2. 確認手順

2-1. マウント状況確認

 docker inspect コンテナ名

Mountsセクションを確認します。

2-2. コンテナ内部確認

 docker exec -it コンテナ名 sh 
 ls -la

マウント先ディレクトリにファイルが存在するか確認します。

2-3. Volume一覧確認

 docker volume ls

不要なVolumeが大量に作成されていないか確認します。

2-4. Volume詳細確認

 docker volume inspect volume名

実際の保存場所を確認できます。

3. 正しい設定例

docker runの場合

 docker run \ -v $(pwd)/data:/app/data \ sample-app

ホスト側のdataディレクトリをコンテナへマウントします。

Docker Composeの場合

 services: app: image: sample-app volumes: - ./data:/app/data

ローカル開発では最もよく利用される設定です。

Named Volume利用例

 services: mysql: image: mysql:8 volumes: - mysql-data:/var/lib/mysql volumes: mysql-data:

データベースの永続化にはNamed Volumeを推奨します。

4. よくあるエラーと対処法

ファイルが反映されない

ホスト側で変更したファイルがコンテナに反映されない場合があります。

原因:

  • パス指定ミス
  • キャッシュ
  • 別ディレクトリを参照している

docker inspectでMountsを確認してください。

Permission denied

 Permission denied

権限不足です。

 chmod -R 755 data

Linux環境ではファイル権限を確認します。

Volumeが消える

docker compose down実行後にデータが消える場合があります。

匿名Volume利用時によく発生します。

Named Volumeへ変更してください。

Mountsが空になる

マウント元ディレクトリが存在しない場合に発生します。

 mkdir -p data

事前にディレクトリを作成してください。

Windows環境でマウントできない

Docker Desktopの共有設定不足が原因です。

  • Settings
  • Resources
  • File Sharing

対象ドライブが共有対象になっているか確認してください。

MySQLデータが保存されない

MySQLコンテナ再作成時にデータが消える場合はVolume設定が不足しています。 

 volumes: - mysql-data:/var/lib/mysql

データディレクトリを永続化してください。

VolumeとBind Mountの違い

種類 用途
Bind Mount 開発環境向け
Named Volume 本番環境向け

開発中はBind Mount、本番ではNamed Volumeを利用するケースが一般的です。

5. まとめ

  • docker inspectでMountsを確認する
  • パス指定ミスを疑う
  • 権限設定を確認する
  • MySQLはNamed Volumeで永続化する
  • Windows環境では共有設定を確認する
