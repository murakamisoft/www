Dockerコンテナが起動しない原因と解決方法

Dockerコンテナが起動しない、すぐ停止する、Exited状態になる場合の原因と解決方法を解説します。 Docker初心者から実務利用まで役立つトラブルシューティング手順をまとめています。

1. Dockerコンテナが起動しない主な原因

Dockerコンテナが起動しない場合、多くはコンテナ自体の問題ではなく、 コンテナ内で動作するアプリケーションや設定ファイルに原因があります。 特にSpring BootやMySQLを利用した開発環境では、設定ミスや接続エラーによって コンテナが即終了（Exited状態）になるケースが頻発します。

原因 発生頻度 内容
アプリケーションエラー ★★★★★ Spring BootやNode.jsが異常終了している
環境変数不足 ★★★★☆ DB接続情報などが設定されていない
ポート競合 ★★★★☆ 同じポートを別プロセスが利用している
Volume設定ミス ★★★☆☆ ファイルのマウントに失敗している
Dockerfile設定ミス ★★★☆☆ CMDやENTRYPOINTが誤っている

2. コンテナ状態の確認方法

まずはコンテナがどのような状態になっているのか確認します。

2-1. コンテナ一覧を確認する

 docker ps -a

出力例です。

 CONTAINER ID IMAGE STATUS 123456789abc sample-app Exited (1)

Exitedになっている場合は、コンテナ内のアプリケーションが異常終了しています。

2-2. ログを確認する

 docker logs コンテナ名

ほとんどの問題はログを見ることで原因を特定できます。

2-3. リアルタイムでログを確認する

 docker logs -f コンテナ名

起動直後に停止する場合はリアルタイム監視が有効です。

3. 原因別の解決方法

3-1. Spring Bootが起動に失敗している

 APPLICATION FAILED TO START

application.ymlやapplication.propertiesの設定ミス、 DB接続エラーが考えられます。

3-2. ポート競合が発生している

 Bind for 0.0.0.0:8080 failed port is already allocated

既に8080ポートを別プロセスが利用しています。

 netstat -ano | findstr 8080

使用中のプロセスを確認してください。

3-3. 環境変数が不足している

 docker inspect コンテナ名

環境変数が正しく設定されているか確認します。

3-4. イメージを再作成する

 docker build --no-cache -t sample .

キャッシュされたイメージが原因の場合があります。

3-5. Docker Compose利用時の再作成

 docker compose down docker compose up --build

コンテナやネットワークを再作成することで解決することがあります。

4. よくあるエラーと対処法

Exited (1)

 Exited (1)

アプリケーション例外による終了です。 ログを確認してください。

Exited (137)

 Exited (137)

メモリ不足によりOSから強制終了された可能性があります。 Docker Desktopのメモリ設定を見直してください。

Exited (143)

 Exited (143)

正常停止時にも発生します。 コンテナ停止操作を実施していないか確認してください。

Restarting状態を繰り返す

 Restarting (1)

起動後すぐに異常終了している状態です。 docker logsで原因を調査してください。

ContainerCreatingのまま進まない

Volumeマウントやイメージ取得に失敗している可能性があります。 Docker Desktopの再起動も有効です。

5. まとめ

  • まず docker ps -a で状態を確認する
  • docker logs で原因を調査する
  • Exited (1) はアプリケーションエラーが多い
  • Exited (137) はメモリ不足を疑う
  • ポート競合や環境変数設定も確認する
  • Docker Compose利用時は再作成を試す
