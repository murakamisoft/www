git resetとgit revertの違い

Gitで「間違えたコミットを取り消したい」と思った時、git resetとgit revertのどちらを使えば良いか迷ったことはありませんか？この2つのコマンドは、取り消し方が全く異なります。

1. 概要：それぞれの動作原理

簡単に言うと、resetは「歴史を書き換える（戻す）」のに対し、revertは「逆の変更を追加する」という違いがあります。

項目 git reset git revert
仕組み ブランチの指し示す位置を強制移動 取り消し内容を新しいコミットとして追加
安全性 低い（履歴が消える） 高い（履歴が残る）

2. 使い分けの基準

チーム開発においては、以下の基準で使い分けるのが基本です。

  • git reset: 自分しか触っていないローカルブランチで、履歴を綺麗にしたい時に使用。
  • git revert: 既にGitHub等へプッシュ済みで、他のメンバーと共有しているコミットを取り消したい時に使用。

※既にプッシュしたコミットに対してresetを使うと、チーム全員の履歴に不整合が起きるため絶対禁止です。

3. コマンドの使い方

3-1. git reset の使用例


# 直前のコミットを取り消して、変更はステージングに残す
git reset --soft HEAD~1

# 直前のコミットを完全に抹消する（危険）
git reset --hard HEAD~1

3-2. git revert の使用例


# 指定したコミット内容を打ち消す新しいコミットを作成する
git revert [コミットハッシュ]

4. よくある失敗と注意点

一番の注意点は、共有ブランチでresetを使ってしまうことです。

もしgit pushした後にresetをして再度pushしようとすると、他のメンバーの履歴と競合してしまい、非常に複雑なトラブルを招きます。公開済みのコミットに対しては、必ずrevertを選択するようにしましょう。

5. まとめ

  • resetはローカルの履歴を「やり直す」ためのもの。
  • revertは共有された履歴を安全に「打ち消す」ためのもの。
  • プッシュ済みのコミットには必ずrevertを使う。
