git push rejectedエラーの原因と対処法

git pushを実行した際に「rejected」エラーが発生して作業が止まってしまったことはありませんか？これはリモートリポジトリがローカルよりも進んでいる場合に発生します。本記事では、この状況を安全に解決する手順を解説します。

1. 概要：pushが拒否される理由

プッシュが拒否される主な原因は、ローカルリポジトリがリモートリポジトリの最新の状態に追いついていないからです。他のメンバーが先にプッシュしており、そのままプッシュすると相手の変更を上書きしてしまう恐れがあるため、Gitは安全のために操作を拒否します。

項目 内容
発生エラー error: failed to push some refs to...
根本原因 リモート側に自分にはないコミットが存在する

2. 手順：リモートの変更を取り込む

基本的には「リモートの変更を先に取得（pull）して統合し、それから自分の作業をプッシュする」という手順を踏みます。 


# 1. リモートの最新情報を取得して統合する
git pull origin [ブランチ名]

# 2. コンフリクトが発生した場合は修正してコミットする

# 3. 再度プッシュする
git push origin [ブランチ名]

3. 修正のポイント

git pullを行った際、コンフリクトが発生した場合は落ち着いて以下の作業を行ってください。

  • 対象ファイルをエディタで開き、競合箇所を修正。
  • git addで修正したファイルをステージング。
  • git commitで統合用のコミットを作成。

4. 注意点：force pushは避ける

ネット上の記事でgit push -f（強制プッシュ）が紹介されていることがありますが、チーム開発では原則禁止です。

理由: 強制プッシュはリモートの歴史を無理やり書き換えるため、他のメンバーの作業内容を消滅させたり、リポジトリ全体を壊したりするリスクがあるからです。必ずpullをして統合してからプッシュする習慣をつけましょう。

5. まとめ

  • push rejectedは、リモートの方が進んでいる時に発生する。
  • まずgit pullを行い、差分を統合する。
  • 強制プッシュ（-f）は使わないこと。
