git cherry-pickが失敗する原因と対処法

Gitで特定コミットだけを別ブランチへ取り込むcherry-pickですが、コンフリクトやコミット未取得などで失敗することがあります。本記事では代表的な原因と解決方法を実例付きで解説します。

1. git cherry-pickとは

git cherry-pickは、別ブランチに存在する特定コミットだけを現在のブランチへ取り込む機能です。

開発現場では以下のような場面でよく利用されます。

  • 障害修正だけを本番ブランチへ反映したい
  • 一部機能のみ別ブランチへ移植したい
  • 誤ったブランチで作業したコミットを移動したい
項目 内容
コマンド git cherry-pick
用途 特定コミットのみ取り込む
利用頻度 高い
トラブル発生率 コンフリクト時に高い

2. 失敗時の確認手順

2-1. コミットが存在するか確認

指定したコミットIDが存在しない場合、cherry-pickは失敗します。


git log --oneline --all

対象コミットが表示されることを確認してください。

2-2. リモート最新情報を取得

他メンバーのコミットを取り込む場合はfetchを実行します。


git fetch origin
git log origin/develop --oneline

2-3. 作業ブランチを確認


git branch

現在のブランチを確認し、正しいブランチ上でcherry-pickを実行してください。

3. cherry-pick実行例

例えばコミット「abc1234」を現在のブランチへ取り込む場合は以下を実行します。 


git cherry-pick abc1234

複数コミットをまとめて取り込む場合は範囲指定も可能です。


git cherry-pick abc1234..def5678

正常終了すると新しいコミットとして現在のブランチへ反映されます。

4. よくあるエラーと対処法

fatal: bad revision


fatal: bad revision 'abc1234'

原因はコミットIDの誤り、またはfetchしていないことです。

以下を実行してコミットを確認してください。


git fetch origin
git log --all --oneline

CONFLICT (content)


CONFLICT (content): Merge conflict in src/Main.java

同じファイルの同じ箇所を別ブランチで変更している場合に発生します。

コンフリクトを解消後、以下を実行します。


git add .
git cherry-pick --continue

cherry-pickを中止したい


git cherry-pick --abort

作業開始前の状態へ戻せます。

マージコミットを指定した場合


error: commit is a merge but no -m option was given

マージコミットには親コミット指定が必要です。


git cherry-pick -m 1 COMMIT_ID

通常は親番号1を指定しますが、プロジェクト運用ルールを確認してください。

空コミットになった場合


The previous cherry-pick is now empty

既に同じ変更が取り込まれている場合に発生します。

内容を確認し、問題なければskipします。


git cherry-pick --skip

5. まとめ

  • cherry-pickは特定コミットのみを取り込める便利な機能
  • 失敗時はコミット存在確認とfetchを最初に実施する
  • コンフリクト発生時は解消後に--continueを実行する
  • 取り消しは--abort、スキップは--skipを利用する
  • マージコミットは-mオプションが必要になる
